Project Objective & Schema Definition

What You're Aiming For

The objective of this project is to practice advanced SQL queries using a pre-defined database schema. This exercise focuses on querying, filtering, aggregating, and manipulating data based on real-world business scenarios. The provided schema models a commercial database that includes categories, customers, orders, employees, suppliers, and products.

The database is characterized by the following relational schema:

Solution Roadmap Strategy

1
Identify Core Entities: Determine which tables contain the required data. If data spans multiple tables, identify the common keys required to JOIN them.
2
Apply Base Filters: Use the WHERE clause early to filter down rows (e.g., date ranges, countries, specific text patterns).
3
Group & Aggregate: When calculating totals or counts, group the data using GROUP BY and filter aggregated results using HAVING.
4
Format Output: Use conditional logic (CASE) and format columns in the SELECT clause. Sort via ORDER BY.

Query 1: High Earning Male Employees

Objective: Display in descending order of seniority the male employees whose net salary (salary + commission) is greater than or equal to 8000.

Explanation: We use the TITLE column to filter for "Mr." to represent male employees. Because the COMMISSION column contains NULL values, doing simple addition (SALARY + COMMISSION) would result in a NULL. We use COALESCE to treat NULL commissions as zero. We use standard EXTRACT functions to calculate age and seniority on the fly based on the CURRENT_DATE.

SELECT EMPLOYEE_int AS "Employee Number", FIRST_NAME || ' ' || LAST_NAME AS "Full Name", EXTRACT(YEAR FROM CURRENT_DATE) - EXTRACT(YEAR FROM BIRTH_DATE) AS "Age", EXTRACT(YEAR FROM CURRENT_DATE) - EXTRACT(YEAR FROM HIRE_DATE) AS "Seniority" FROM EMPLOYEES WHERE TITLE = 'Mr.' AND (SALARY + COALESCE(COMMISSION, 0)) >= 8000 ORDER BY "Seniority" DESC;

Query 2: Advanced Product Filtering

Objective: Display products meeting 5 strict criteria: (C1) packaged in bottles, (C2) 3rd character is 't' or 'T', (C3) suppliers 1,2,3, (C4) price 70-200, (C5) units ordered is not null.

Explanation: This requires combining multiple string manipulation and logical operators. For C1, LIKE '%bottle%' acts as a wildcard search. For C2, the pattern __t% specifies exactly two wildcard characters before the 't'. We use IN for the supplier list, and BETWEEN for the pricing range.

SELECT PRODUCT_REF, PRODUCT_NAME, SUPPLIER_int, UNITS_ON_ORDER, UNIT_PRICE FROM PRODUCTS WHERE QUANTITY LIKE '%bottle%' AND (PRODUCT_NAME LIKE '__t%' OR PRODUCT_NAME LIKE '__T%') AND SUPPLIER_int IN (1, 2, 3) AND UNIT_PRICE BETWEEN 70 AND 200 AND UNITS_ON_ORDER IS NOT NULL;

Query 3: Customers in Supplier 1's Region

Objective: Display customers who reside in the same region as supplier 1 (same country, city, and last three digits of postal code) using a single subquery.

Explanation: To evaluate multiple columns using a single subquery, we utilize a Tuple Subquery (or Row Constructor). We parse the right side of the postal code string using the RIGHT() function and compare all three aspects simultaneously.

SELECT * FROM CUSTOMERS WHERE (COUNTRY, CITY, RIGHT(POSTAL_CODE, 3)) = ( SELECT COUNTRY, CITY, RIGHT(POSTAL_CODE, 3) FROM SUPPLIERS WHERE SUPPLIER_int = 1 );

Query 4: Order Discount Categorization

Objective: Analyze specific order numbers and output a dynamic discount rate string alongside a logic application note using defined brackets.

Explanation: This is a complex query requiring GROUP BY and dual CASE statements. Since an order can consist of multiple lines in the ORDER_DETAILS table, we must aggregate them first using SUM(UNIT_PRICE * QUANTITY). The CASE statements act as IF/THEN/ELSE logic trees inside the SQL query itself.

SELECT ORDER_int, CASE WHEN SUM(UNIT_PRICE * QUANTITY) BETWEEN 0 AND 2000 THEN '0%' WHEN SUM(UNIT_PRICE * QUANTITY) BETWEEN 2001 AND 10000 THEN '5%' WHEN SUM(UNIT_PRICE * QUANTITY) BETWEEN 10001 AND 40000 THEN '10%' WHEN SUM(UNIT_PRICE * QUANTITY) BETWEEN 40001 AND 80000 THEN '15%' ELSE '20%' END AS "New Discount Rate", CASE WHEN ORDER_int BETWEEN 10000 AND 10999 THEN 'apply old discount rate' ELSE 'apply new discount rate' END AS "Discount Rate Note" FROM ORDER_DETAILS WHERE ORDER_int BETWEEN 10998 AND 11003 GROUP BY ORDER_int;

Query 5: Suppliers of Beverage Products

Objective: Bridge tables to identify the suppliers providing goods under a specific category.

Explanation: This demonstrates a typical 3-table JOIN traversing from Suppliers to Products to Categories. We use DISTINCT because a supplier might provide 5 different beverages, and without it, their company name would print 5 times.

SELECT DISTINCT S.SUPPLIER_int, S.COMPANY, S.ADDRESS, S.PHONE FROM SUPPLIERS S JOIN PRODUCTS P ON S.SUPPLIER_int = P.SUPPLIER_int JOIN CATEGORIES C ON P.CATEGORY_CODE = C.CATEGORY_CODE WHERE C.CATEGORY_NAME = 'Beverages';

Query 6: Berlin Customers with ≤ 1 Dessert

Objective: Display Berlin customers who have ordered at most 1 dessert, including those who ordered 0.

Explanation: The requirement "at most 1 (0 or 1)" is critical. If we use an INNER JOIN, customers with 0 dessert orders are excluded immediately. Therefore, we chain LEFT JOINs. In the HAVING clause, we conditionally count only the products that fall under the 'Desserts' category to ensure we aren't counting their non-dessert orders.

SELECT C.CUSTOMER_CODE FROM CUSTOMERS C LEFT JOIN ORDERS O ON C.CUSTOMER_CODE = O.CUSTOMER_CODE LEFT JOIN ORDER_DETAILS OD ON O.ORDER_int = OD.ORDER_int LEFT JOIN PRODUCTS P ON OD.PRODUCT_REF = P.PRODUCT_REF LEFT JOIN CATEGORIES CAT ON P.CATEGORY_CODE = CAT.CATEGORY_CODE WHERE C.CITY = 'Berlin' GROUP BY C.CUSTOMER_CODE HAVING COUNT(DISTINCT CASE WHEN CAT.CATEGORY_NAME = 'Desserts' THEN P.PRODUCT_REF END) <= 1;

Query 7: French Customers' Monday Orders in April 1998

Objective: Total amounts for Monday orders in a specific month, ensuring customers with NO orders in that timeframe are still listed.

Explanation: Similar to Query 6, keeping zero-order customers requires a LEFT JOIN. Crucially, the filtering logic (Year, Month, Day) must be placed inside the ON clause of the join, rather than the WHERE clause, to prevent it from filtering out the base customer rows. *(Syntax note: `DAYNAME()` is used in MySQL. Postgres uses `TO_CHAR(date, 'Day')`.)*

SELECT C.CUSTOMER_CODE, C.COMPANY, C.PHONE, COALESCE(SUM(OD.UNIT_PRICE * OD.QUANTITY), 0) AS "Total Amount", C.COUNTRY FROM CUSTOMERS C LEFT JOIN ORDERS O ON C.CUSTOMER_CODE = O.CUSTOMER_CODE AND EXTRACT(YEAR FROM O.ORDER_DATE) = 1998 AND EXTRACT(MONTH FROM O.ORDER_DATE) = 4 AND DAYNAME(O.ORDER_DATE) = 'Monday' LEFT JOIN ORDER_DETAILS OD ON O.ORDER_int = OD.ORDER_int WHERE C.COUNTRY = 'France' GROUP BY C.CUSTOMER_CODE, C.COMPANY, C.PHONE, C.COUNTRY;

Query 8: Customers Who Have Ordered ALL Products

Objective: Find users whose order history covers the entirety of the product catalog.

Explanation: This constraint is known as "Relational Division". Instead of attempting massive loop logic, we simply count the number of DISTINCT products a customer has purchased and check if that integer matches the absolute count of products available in the database.

SELECT C.CUSTOMER_CODE, C.COMPANY, C.PHONE FROM CUSTOMERS C JOIN ORDERS O ON C.CUSTOMER_CODE = O.CUSTOMER_CODE JOIN ORDER_DETAILS OD ON O.ORDER_int = OD.ORDER_int GROUP BY C.CUSTOMER_CODE, C.COMPANY, C.PHONE HAVING COUNT(DISTINCT OD.PRODUCT_REF) = (SELECT COUNT(*) FROM PRODUCTS);

Query 9: Number of Orders per French Customer

Objective: Display customer code and total order count for France.

Explanation: A standard aggregation. We use a LEFT JOIN to the ORDERS table so we don't lose customers with zero orders. We specifically COUNT(O.ORDER_int) rather than COUNT(*) so that NULL rows evaluate to 0 instead of 1.

SELECT C.CUSTOMER_CODE, COUNT(O.ORDER_int) AS "Number of Orders" FROM CUSTOMERS C LEFT JOIN ORDERS O ON C.CUSTOMER_CODE = O.CUSTOMER_CODE WHERE C.COUNTRY = 'France' GROUP BY C.CUSTOMER_CODE;

Query 10: Yearly Order Differences

Objective: Display orders in 1996, 1997, and the mathematical difference between them in a single row.

Explanation: This utilizes a powerful technique called Conditional Aggregation. By placing a CASE statement inside a SUM function, we can pivot data from rows into distinct columns without needing complex subqueries. We wrap it in an outer query to perform the subtraction cleanly.

SELECT SUM(CASE WHEN EXTRACT(YEAR FROM ORDER_DATE) = 1996 THEN 1 ELSE 0 END) AS "Orders in 1996", SUM(CASE WHEN EXTRACT(YEAR FROM ORDER_DATE) = 1997 THEN 1 ELSE 0 END) AS "Orders in 1997", SUM(CASE WHEN EXTRACT(YEAR FROM ORDER_DATE) = 1997 THEN 1 ELSE 0 END) - SUM(CASE WHEN EXTRACT(YEAR FROM ORDER_DATE) = 1996 THEN 1 ELSE 0 END) AS "Difference" FROM ORDERS;
